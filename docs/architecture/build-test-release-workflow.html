Developer Workflow

Build, Test, and Release

Use the narrowest workflow that proves the change. Stay fast while editing, then add trust, runtime, and distribution checks only when the job needs them.

Levels

The Ladder

KeyPath has five practical levels, from quick local iteration to public release artifacts. Installed-app verification is a reusable gate inside the signed lanes.
01

Inner Loop

Fast

Use for Swift, UI, and resource edits while you are still shaping the change. Build first, then run the smallest useful test lane. Deploy only when you need to inspect installed app behavior.

swift build
./Scripts/test-fast.sh --changed
python3 Scripts/check-accessibility.py
02

Local Test Gate

Local

Use before pushing, after focused test filters pass, or when reproducing CI behavior locally. The safe runner builds tests first, supports TEST_FILTER/TEST_SKIP, uses a watchdog, isolates HOME, and skips event-tap paths.

./Scripts/test-full.sh
03

Pull Request CI

Required

Runs on the self-hosted macOS runner. It restores or builds cached kanata fork artifacts, runs the safe test suite, parses the installer matrix from the test log, verifies the SMAppService plist, and enforces code quality. Coverage runs in the separate scheduled/manual KeyPath Coverage workflow.

gh pr checks
./Scripts/test-ci-locally.sh
04

Release Candidate

Signed

Use after merge when /Applications/KeyPath.app should match a real Developer ID and notarized build for manual testing. This is the normal signed local deployment lane.

git fetch --prune origin
git pull --ff-only origin master
./Scripts/release-candidate.sh
05

Public Release

Ship

Use only when producing public artifacts. This path can bump versions, notarize and staple the app, create Sparkle and DMG artifacts, tag, publish GitHub releases, update appcast, publish website content, and update Homebrew.

./Scripts/release-doctor.sh --ship
./Scripts/release.sh <version>
Which Lane

Choosing Commands

Most changes should climb the ladder gradually. Do not run a notarized public release path just to check a SwiftUI edit.
One Swift file
Run ./Scripts/test-fast.sh --changed, ./Scripts/test-fast.sh <area>, or TEST_FILTER=SomeTests ./Scripts/run-tests-safe.sh, then swift build.
UI iteration
Run filtered UI tests first. Use ./Scripts/quick-deploy.sh only when installed app behavior needs inspection. Use Poltergeist only for focused single-agent UI work, then stop it.
Helper or service
Avoid hot-swapping the helper by accident. Use installer-focused tests and the release-candidate lane when the installed runtime matters.
Before push
Run ./Scripts/test-full.sh. The pre-push hook uses this same safe path.
After merge
Deploy from the intended master worktree with ./Scripts/release-candidate.sh.
Public ship
Run ./Scripts/release-doctor.sh --ship, then ./Scripts/release.sh <version>. Public releases must be notarized.
Quality Gates

What Each Gate Proves

Each level adds a different kind of evidence. The goal is to catch cheap failures early and reserve slow checks for places where their evidence matters.
1
Fast build correctness
swift build proves the package still compiles and keeps incremental caches warm.
2
Safe test correctness
run-tests-safe.sh uses CI-safe environment variables and now treats non-zero exits as failures unless an escape hatch is explicitly enabled.
3
PR quality correctness
CI enforces changed-file formatting, SwiftLint, no-sleep guardrails, accessibility checks, and fails on TODO CRITICAL or FIXME URGENT markers.
4
Distribution trust correctness
release-candidate.sh signs, notarizes, deploys, and runs verify-installed-app.sh to check code signature, Gatekeeper, stapling, process launch, launchd, and TCP readiness.
5
Public artifact correctness
release.sh is for user-facing release assets: Sparkle archive, appcast entry, DMG, GitHub release, website, and Homebrew updates.
Do not infer release health from registration alone. A signed deployment is healthy only after trust checks pass and runtime readiness is verified through process and TCP evidence.
Escape Hatches

Use Sparingly

These flags exist for diagnosis. If they become normal workflow, the underlying failure mode should be fixed.
Runner crash pass
KEYPATH_ALLOW_TEST_RUNNER_CRASH_SUCCESS=1 allows a signal crash to pass only when parsed test failures are zero; PR CI uses this for the known SwiftPM-wrapped XCTest post-run abort.
Non-zero test pass
KEYPATH_ALLOW_NONZERO_TEST_SUCCESS=1 restores the old behavior where a non-zero test runner exit can pass if tests were parsed as passing.
Skip RC doctor
./Scripts/release-candidate.sh --no-doctor skips preflight only while debugging the release scripts.
Skip RC verify
./Scripts/release-candidate.sh --no-verify skips installed-app verification. Do not use it for manual release-candidate validation.
Debug trust only
CHECK_RUNTIME=0 ./Scripts/verify-installed-app.sh checks signature and notarization without requiring the app to be running.
