Architecture

Overlay Rendering

How each keycap decides what to show.

Overview

Three Rendering Modes

Every keycap on the overlay renders in exactly one of three modes, determined by the current layer and activation state.

Base

Standard keycap: letters, shift symbols, function keys, modifiers. Handles floating label animation.

Layer

Active when holding a layer key: arrows, SF symbols, action icons, collection-colored backgrounds. Key letter dims to top-left.

Launcher

Active in launcher mode: app icons centered, key letter in top-right corner. Transition animation between states.

The overlay originally rendered all keycap content inside a single 2,685-line OverlayKeycapView. The decomposition into focused child views followed a simple principle: the parent owns the shell (background, shadows, gestures, icon loading), while children own the content (what text, symbol, or icon to draw). Each child view is independently testable and snapshot-verifiable.
Architecture

The Composition Pattern

A parent shell dispatches to one of three child views. The dispatch is a linear if-else chain — no key is ever in two modes at once.
The parent OverlayKeycapView handles everything shared across modes: keycap background color, shadows, glow effects, press animation, selection/hover highlights, gesture handling, and icon loading lifecycle (onAppear/onChange). It then dispatches to the appropriate child view based on the current mode. The parent weighs in at ~1,556 LOC spread across 10 files.
The child views are deliberately small and focused. LauncherModeKeycap is 108 LOC with 11 parameters. LayerModeKeycap is 255 LOC with 14 parameters. BaseKeycap is 852 LOC with 23 parameters (the largest, because base mode handles the most label variants). KeycapSymbols is a 177 LOC shared utility used by all three.
Data Flow

What Each Child Receives

The parent passes only what each child needs. No child has access to the full parent state.

Physical key identifier for launch action dispatch

Key letter shown in top-right corner
holdLabel
Secondary label for tap-hold keys
scale
Keycap scale factor from layout geometry

The LauncherMapping bound to this key, if any

Pre-loaded images from the parent's @State

Animation phase for enter/exit transitions

The full KeyInfo for geometry and identity

Custom SF symbol or image override

Concepts

Key Types

Four types that power the rendering pipeline, each with a single responsibility.
FloatingLabelVisibility Struct
Pure struct that decides whether a floating label is visible. Evaluates 7 conditions as a single isVisible(label) call: is the key a tap-hold key? Is it idle? Is the floating labels preference on? Is the label non-empty? Is the layer base? Is the colorway standard? Is the key not pressed? Testable in isolation with 13 unit tests.
LayerMappingBuilder Pipeline
Pipeline that transforms raw simulator output into display-ready layer mappings. Three steps: augment with push-msg actions, apply app-specific overrides, enrich with custom shift labels. Each step is a pure function that can be tested independently.
KeycapSymbols Utility
Shared utility enum used by all three child views. SF symbol resolution for actions, window action symbols and colors, collection-to-color mapping, modifier key detection, and dynamic text label sizing. 177 LOC of pure functions with no state.
GMKColorway Theme
Colorway theming system. Four legend styles: .standard (letters), .dots (circles), .blank (nothing), .iconMods (symbols for modifiers). Each style routes through a different content path in keyContent, which is why the dispatch tree branches before reaching standardKeyContent.
Principles

Design Rules

Five rules that govern the decomposition. Each exists because the monolithic version violated it and caused bugs.
1
Parent owns the shell, children own the content
Background color, shadows, glow, gestures, and icon loading lifecycle stay in the parent. Content rendering (what text/icon to show) lives in child views.
2
@State stays in the parent
appIcon, faviconImage, launcherTransition are @State on OverlayKeycapView. Children receive them as plain parameters, never as bindings.
3
Children never reference the parent
No child view has a dependency on OverlayKeycapView. They use KeycapSymbols for shared utilities and take all data as init parameters.
4
Snapshot tests gate every change
15 keycap-level + 6 full-overlay snapshot tests verify pixel-identical rendering before and after any structural change.
5
Geometry follows PhysicalLayout, labels follow LogicalKeymap
The layout determines key positions and sizes. The keymap determines what letter to print. No UI toggle mixes them.
Core invariant. Each keycap renders exactly one mode. The parent's standardKeyContent is a linear if-else chain: touchId → launcher → layer → base. A key is never in two modes simultaneously.
