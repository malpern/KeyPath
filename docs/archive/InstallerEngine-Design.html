Method 1

inspectSystem()

func inspectSystem() -> SystemContext

Purpose: Capture current system state

Returns: Read-only snapshot of service states, file/permission status, and helper availability

Use this to:

  • Render current status in UI
  • Decide what actions are needed
  • Provide context for planning

Wraps: Consolidates logic from SystemSnapshotAdapter, SystemRequirements, ServiceStatusEvaluator, and conflict detection scripts (dev-tools/test-updated-conflict.swift) that currently run separately.

Method 2

makePlan()

func makePlan(for intent: InstallIntent, context: SystemContext) -> InstallPlan

Purpose: Create an execution plan without running it

Returns: Ordered list of operations tailored to the observed context. If prerequisites are unmet, the plan will be marked as .blocked with details about missing requirements.

Converts intents into:

  • Required ServiceRecipes
  • Privileged operations
  • Configuration updates
  • Health check steps
  • Requirement validation (admin rights, writable directories, SMAppService approval, helper registration)
üí° Supported Intents
.install, .repair, .uninstall, .inspectOnly

Wraps: Captures orchestration from WizardAutoFixer, LaunchDaemonInstaller (all the create...Service helpers), PackageManager, and health/driver routines in VHIDDeviceManager.

Method 3

execute()

func execute(plan: InstallPlan, using broker: PrivilegeBroker) -> InstallerReport

Purpose: Execute the planned operations

Returns: Structured report with success/failure details and final state. If the plan was blocked by unmet requirements, execution stops immediately and the report indicates which requirement failed.

Performs actions such as:

  • Copy plists to system directories
  • Bootstrap LaunchDaemons
  • Restart unhealthy services
  • Install log rotation
  • Register SMAppService items

Wraps: Becomes a fa√ßade over PrivilegedOperationsCoordinator, HelperManager, Authorization Services fallbacks, and admin-dialog scripts (e.g., dev-tools/test-admin-dialog-direct.swift), while reusing the existing InstallerReport struct.

Method 4

run()

func run(intent: InstallIntent, using broker: PrivilegeBroker) -> InstallerReport

Purpose: Convenience wrapper that chains inspectSystem() ‚Üí makePlan() ‚Üí execute() internally

Useful for CLI "one-button repair" automation or simple GUI flows. The intermediate SystemContext and InstallPlan are still available through logging/callbacks for debugging and inspection.

Wraps: Mirrors today's wizard auto-fix button (WizardAutoFixer.performAutoFix) and CLI scripts that manually chain detection + install‚Äînow it just calls the fa√ßade methods in order.

Advanced Scenarios

üîç Dry Run Mode

CLI can generate a plan and print it without calling execute(), useful for:

  • Previewing changes
  • Debugging issues
  • Documentation

üîÑ Fix Button Flow

GUI can implement a "Fix" button using the convenience method:

  1. Call run(intent: .repair, using: broker) which chains inspect ‚Üí plan ‚Üí execute internally
  2. Check InstallerReport for success/failure
  3. If blocked by requirements, surface blocking items to user
  4. Re-detect with inspectSystem() to show success

Or use the full pipeline for more control: inspectSystem() ‚Üí makePlan() ‚Üí execute()

‚ö†Ô∏è Failed Preconditions

If makePlan() returns a blocked plan:

  • Plan status is .blocked with details about unmet requirements
  • Caller should surface blocking items to user before attempting execution
  • execute() will stop immediately if given a blocked plan

If something fails mid-execution:

  • execute() reports failing step in InstallerReport
  • UI/test decides: retry, prompt for intervention, or abort

  • LaunchDaemonInstaller.getServiceStatus()

Planning

  • LaunchDaemonInstaller (all the create...Service helpers)
  • PackageManager (component installation)
  • Health/driver routines in VHIDDeviceManager
  • createAllLaunchDaemonServices
  • restartUnhealthyServices
  • Config writing routines
  • Version checks (shouldUpgradeKanata)
  • installLogRotationService

Requirement validation (admin rights, writable directories, SMAppService approval, helper registration) is handled during planning‚Äîunmet requirements result in a .blocked plan status.

Execution

  • HelperManager (helper IPC path)
  • Authorization Services fallbacks
  • Admin-dialog scripts (dev-tools/test-admin-dialog-direct.swift)

  • AppleScript fallbacks

Reuses the existing InstallerReport struct from LaunchDaemonInstaller to maintain logging/test continuity.

Privilege Handling
