# docs/ reorganization plan

Status: **proposal — awaiting approval**. No files moved yet.

## Problem

`docs/` has **38 loose `.md` files** at its root, mixing user guides, dev
references, test plans, research, and ephemeral dashboards — while well-formed
subdirs (`adr/`, `architecture/`, `features/`, `testing/`) already sit right
next to them. Two naming conventions clash: `SCREAMING_SNAKE_CASE.md` and
`kebab-case.md`. There are also two near-identical dirs: `docs/plans/` and
`docs/planning/`.

**Constraint:** 21 doc paths are referenced from `CLAUDE.md`, `README.md`,
`CONTRIBUTING.md`, `AGENTS.md`, plus Swift/shell source. Every move must update
those inbound links (esp. CLAUDE.md's "Deep-Dive References" table).

## Decisions to lock before executing

1. **Naming convention:** standardize on **`kebab-case.md`** (matches newer files
   and gh-pages guides). Requires `git mv` on ~20 SCREAMING_SNAKE files.
2. **plans vs planning:** merge into a single **`docs/planning/`** (living plans);
   completed plans → `docs/archive/`.
3. **Ephemeral dashboards:** `RELEASE-READINESS.md`, `RELEASE-READINESS.html`,
   `CODE_AUDIT.md` are point-in-time snapshots → move to `docs/archive/` (or
   delete if regenerated by scripts — verify first).

## Execution (phased, one PR)

1. `git mv` files into target dirs with kebab-case names (preserves history).
2. Merge `plans/` → `planning/`; archive completed plans.
3. Move the 3 ephemeral dashboards to `archive/`.
4. Grep-and-replace all 21 inbound references (CLAUDE.md table, README, AGENTS,
   CONTRIBUTING, source, scripts). Verify with a link-check pass.
5. Add a one-line `README.md` to each new subdir describing its scope.
6. Run `./Scripts/test-fast.sh --changed` (docs-only, but confirms no script
   references broke) and open PR.

## Risk / notes

- Pure doc move — no runtime impact. Main risk is a missed inbound link; mitigated
  by the grep pass in step 4.
- gh-pages `guides/` is a *separate* publishing system (per CLAUDE.md) — this reorg
  touches only `docs/` (developer-facing), not the published user guides. Naming
  `docs/guides/` here is internal and won't collide with gh-pages.
